class _SliverAppBarDelegate extends SliverPersistentHeaderDelegate {
  _SliverAppBarDelegate({
    required this.topPadding,
    required this.bottomHeight,
    required this.child,
  });

  final double topPadding;
  final double bottomHeight;
  final Widget child;

  @override
  double get minExtent => topPadding + kToolbarHeight;

  @override
  double get maxExtent => topPadding + kToolbarHeight + bottomHeight;

  @override
  Widget build(
      BuildContext context, double shrinkOffset, bool overlapsContent) {
    return SizedBox.expand(child: child);
  }

  @override
  bool shouldRebuild(_SliverAppBarDelegate oldDelegate) {
    return child != oldDelegate.child ||
        topPadding != oldDelegate.topPadding ||
        bottomHeight != oldDelegate.bottomHeight;
  }
}

The _SliverAppBarDelegate class is used to create a delegate for the SliverPersistentHeader widget. It takes in three parameters: topPadding, bottomHeight, and child. The topPadding parameter is the amount of padding at the top of the SliverAppBar, the bottomHeight parameter is the height of the TabBar, and the child parameter is the SliverAppBar widget itself.

The build method of the _SliverAppBarDelegate class returns a SizedBox.expand widget with the child SliverAppBar widget. This is necessary because the build method of a SliverPersistentHeader delegate must return a widget that takes up the full extent of the header.

The shouldRebuild method is used to determine whether the delegate should be rebuilt when the SliverAppBar is updated. In this case, it compares the child, topPadding, and bottomHeight parameters to determine if the delegate needs to be rebuilt.

To achieve the behavior you described, you can use the NestedScrollView and TabLayout widgets provided by the Android framework. Here's an example layout file that demonstrates how to implement this:

ini
<androidx.coordinatorlayout.widget.CoordinatorLayout
    android:layout_width="match_parent"
    android:layout_height="match_parent">

    <com.google.android.material.appbar.AppBarLayout
        android:layout_width="match_parent"
        android:layout_height="wrap_content">

        <androidx.appcompat.widget.Toolbar
            android:id="@+id/toolbar"
            android:layout_width="match_parent"
            android:layout_height="?attr/actionBarSize"
            app:layout_scrollFlags="scroll|enterAlways"/>

        <com.google.android.material.tabs.TabLayout
            android:id="@+id/tab_layout"
            android:layout_width="match_parent"
            android:layout_height="wrap_content"
            app:layout_scrollFlags="scroll|enterAlways"/>

    </com.google.android.material.appbar.AppBarLayout>

    <androidx.core.widget.NestedScrollView
        android:layout_width="match_parent"
        android:layout_height="match_parent"
        app:layout_behavior="@string/appbar_scrolling_view_behavior">

        <!-- Your scrollable content here -->

    </androidx.core.widget.NestedScrollView>

</androidx.coordinatorlayout.widget.CoordinatorLayout>

In this layout, the AppBarLayout contains a Toolbar and a TabLayout. Both of these views have the scroll and enterAlways scroll flags set, which means that they will scroll off-screen when the user scrolls down, but will reappear when the user scrolls up.

The NestedScrollView contains your scrollable content and has the appbar_scrolling_view_behavior set as its layout behavior. This tells the CoordinatorLayout to handle scrolling events in a way that makes the AppBarLayout and its children sticky.

You can customize this layout to fit your specific needs, but this should give you a good starting point.
